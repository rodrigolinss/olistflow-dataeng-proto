Por que um marketplace brasileiro?

R$ 185 bi E-commerce Brasil · 2024
Marketplaces concentram a maior complexidade de dados

Cada pedido envolve cliente, vendedor, meio de pagamento e operador logístico — em todos os estados, com ciclos longos da compra à avaliação. Os dados nascem dispersos em sistemas que não conversam.

Três dores recorrentes

  • Visão fragmentada — cada equipe consulta uma fonte diferente
  • Decisões lentas — "qual categoria cresceu?" leva dias
  • Sinais perdidos — navegação nunca cruza com pós-compra
O OlistFlow ataca essas três dores com um pipeline end-to-end unificando dados operacionais (batch) e comportamentais (streaming).
Contextualizar sem jargão. Tempo: 1,5 min. Frase-âncora: "marketplaces são complexos porque os dados nascem dispersos em sistemas desacoplados e ninguém tem a visão completa da jornada."

Tratamos os dois porque contam histórias diferentes da mesma jornada.

Tempo: 2,5 min. Focar na dicotomia: batch = estado do negócio; streaming = comportamento do usuário. Destacar que a gente usa dataset público REAL (Olist) e simula eventos porque não existe clickstream público acoplado.

Tempo: 2 min. Pontos-chave: (1) separação batch/streaming em DAGs diferentes; (2) dbt_tests + GE são gates — falha crítica interrompe antes do refresh Metabase (dado ruim nunca vira dashboard); (3) idempotência por sobrescrita de partição permite religar qualquer DAG sem duplicar.

Por que Medalhão e não outras?

ArquiteturaPor que não ganhou
Data Warehouse puroCarrega dados já transformados — perde raw, não reprocessa. Inadequado para eventos semi-estruturados.
Data Lake puroSem estrutura vira "pântano de dados". Não cobre confiabilidade exigida.
Lambda puro (Kafka + Spark)Overkill. Duas stacks completas exigem hardware inviável em contexto acadêmico.
Kappa puroTrata tudo como streaming. Desproporcional — maior parte dos dados é batch por natureza.
Data MeshExige múltiplos times com ownership. Uma dupla de estudantes não reproduz isso.
Medalhão ✅Vencedor: cobre requisitos, roda em notebook, expressa princípios de camadas/acoplamento/reversibilidade.

🪶 Cabe no hardware

DuckDB in-process · Parquet local · sem cluster

⏪ Reversibilidade

Bronze imutável · qualquer bug reconstrói tudo

🚀 Migrável

Troca DuckDB→BigQuery e Parquet→S3 sem redesenho

Essa tabela mostra que consideramos alternativas com seriedade. Destacar: Medalhão não é "escolha preguiçosa" — é equilíbrio entre literatura moderna e viabilidade. Se perguntarem "e Data Mesh?", responder: "nossa separação por domínios já prepara, é só empacotar se o projeto crescer".

Três decisões não-triviais

🦆 DuckDB
sobre Spark

Pra 45 MB de dados, Spark seria overkill de ordem de grandeza.

DuckDB in-process. SQL ANSI. Integração direta com dbt.

⚡ Redis Streams
sobre Kafka

Kafka exige ZooKeeper + brokers + 2 GB só em overhead.

Redis tem a mesma semântica de log imutável em 50 MB.

🛠 dbt
como espinha

Transforma SQL em código versionável com testes e linhagem automática.

É o padrão de mercado hoje — vale aprender.

Cada escolha tem plano B documentado no relatório. Se Redis não rodar na sala, o simulador grava direto em JSONL — mesmo pipeline, menos infra.
Tempo: 2 min. Mostrar que raciocinamos, não copiamos. Destacar o fallback do Redis → JSONL como exemplo de pragmatismo (não dogmatismo). Se perguntarem "por que não Postgres direto?": "Postgres é OLTP, não escala pra analytics columnar — DuckDB é colunar, feito pra isso."

Tradeoffs honestos

✅ O que ganhamos

  • Baixo acoplamento: camadas conversam via Parquet com schema contratado
  • Confiabilidade: dbt tests + GE bloqueiam dados inválidos
  • Reversibilidade: Bronze imutável permite --full-refresh
  • Linhagem automática: dbt gera grafo completo
  • Portabilidade: mesma topologia roda em S3/BigQuery

⚠️ O que cedemos conscientemente

  • Alta disponibilidade real: single-node, falha = downtime
  • Streaming industrial: Redis < Kafka; volumes didáticos
  • RBAC granular no serving: Metabase com auth simples
  • Data catalog corporativo: usamos dbt docs local
  • Escala horizontal: DuckDB é single-node

Todas as renúncias são recuperáveis em uma versão futura — sem alterar a topologia principal.

Tempo: 1,5 min. Honestidade vende mais que vender o produto perfeito. Deixar claro: "cedemos porque é acadêmico; em produção seria diferente — mas a arquitetura que escolhemos permite essa evolução sem redesenho."
